# Backend Implementation Proposal: Violation Logging & Heartbeat API (v2)

This document outlines the backend architecture to support the periodic proctoring heartbeat and violation telemetry.

## 1. Database Schema

### `ExamHeartbeat`
Stores the periodic (5-minute) consolidated telemetry bucket.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key. |
| `exam_access` | ForeignKey | Links to `vmlc.ExamAccess`. |
| `sequence_number` | PositiveInt | Sequential index of the heartbeat (e.g., 1, 2, 3...) to detect gaps. |
| `client_uuid` | UUID | Idempotency key generated by the client to prevent duplicates on retries. |
| `timestamp` | DateTime | Server-side receipt time. |
| `period_start` | DateTime | Start of the 5-minute interval (Client-side). |
| `period_end` | DateTime | End of the 5-minute interval (Client-side). |
| `summary` | JSONField | Map of event types to counts (e.g., `{"TAB_SWITCH": 2}`). |
| `face_capture` | ImageField | Representative frame (stored in PrivateMediaStorage). |
| `suspicion_score` | Float | Computed score for this specific bucket (0.0 - 1.0). |
| `meta` | JSONField | Client environment info (Browser, OS, Screen Resolution). |

### `ViolationEvent`
Stores individual high-fidelity events captured within a heartbeat period.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key. |
| `heartbeat` | ForeignKey | Links to `ExamHeartbeat`. |
| `event_type` | CharField | `TAB_SWITCH`, `MULTI_FACE`, `NO_FACE`, `DEVTOOLS_OPEN`, etc. |
| `timestamp` | DateTime | Precise time the event occurred (Client-side). |
| `is_critical` | Boolean | Whether this event should trigger immediate staff notification. |
| `metadata` | JSONField | Context (e.g., `question_id`, `duration_seconds`, `snapshot_data`). |

## 2. API Design Improvements

### Endpoint: `POST /v2/exams/{exam_id}/heartbeat/`

**Idempotency & Reliability:**
- The client MUST provide a `client_uuid`. If the backend receives a duplicate UUID for the same `ExamAccess`, it returns `200 OK` but skips processing.
- **Gap Detection:** If `sequence_number` arrives out of order (e.g., 1 then 3), the backend flags a "Telemetry Gap" in the proctoring log.

**Security:**
- **Signed URLs for Metadata:** If the client needs to report a violation that includes a sensitive snapshot, the backend can provide temporary S3 upload slots, though for heartbeats, `multipart/form-data` is preferred.

---

## 3. Automated Suspicion Scoring (Service Layer)

The `ProctoringService` will calculate a `suspicion_score` for every heartbeat:
- **Base Weights:** 
    - `TAB_SWITCH`: 0.3
    - `MULTI_FACE`: 0.5
    - `FULLSCREEN_EXIT`: 0.1
    - `NO_FACE` (> 30s): 0.2
- **Thresholds:** If a single heartbeat score > 0.8, the `ExamAccess` is automatically flagged as "Highly Suspicious".

## 5. Data Retention & Privacy
- **Face Captures:** Purged 30 days after the competition concludes unless the candidate is "Flagged".
- **Logs:** Retained for 1 year for audit purposes.
